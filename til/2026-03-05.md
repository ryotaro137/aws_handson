## AWS Control Tower とランディングゾーン
- **概要**: 複数アカウント環境（ランディングゾーン）をベストプラクティスに基づき自動構築・管理するマネージドサービス。ガードレールを用いてガバナンスを効かせる。
- **試験ポイント**:
    - **ガードレールの種類**: 予防的（SCP使用、APIレベルで拒否）と発見的（Config使用、違反を検知）の使い分け。
    - **リージョン制限**: Control Towerの設定でガバナンス対象リージョンを指定し、組織全体で標準化された構成を維持できる。
    - **Account Factory**: 新規アカウント作成時に自動でガードレールやベースラインを適用する仕組み。
- **実務での使い方**: 
    - **マルチアカウント戦略**: 本番、開発、共有サービス、ログ保管用など、アカウントを用途別に分離し、Control Towerで一元管理する「Landing Zone」の構築に必須。
    - **コンプライアンス監視**: ダッシュボードで組織全体の非準拠リソースを可視化し、一元的な監視体制を構築する。
- **副業での需要**: ★★★★ (セキュリティ運用・ガバナンス強化)。新規プロジェクトの立ち上げ時に「安全なマルチアカウント環境をサクッと作りたい」という需要が多い。初期構築パッケージ（30〜60万円〜）や継続的なガバナンス支援で高く評価される。

## Service Control Policy (SCP) による組織統制
- **概要**: AWS Organizationsで組織単位（OU）やアカウントに適用できる最大権限の境界（Permission Boundary）を定義するポリシー。
- **試験ポイント**:
    - **権限の制限**: 管理者（AdministratorAccess）であっても、SCPで拒否されたアクションは実行できない（APIレベルでの予防的制御）。
    - **リージョン制御**: `aws:RequestedRegion` 条件キーを使用して、特定のリージョン以外へのアクセスを遮断できる。Control Towerと統合することで自動適用が可能。
    - **継承**: ルート → OU → アカウントの順に適用され、上位で拒否されたものは下位で許可できない。
- **実務での使い方**: 
    - **代替アプローチの課題解決**: IAMレベルでの個別制御やConfigによる事後検知に比べ、管理が容易で「リソース作成を未然に防ぐ」ためコスト・運用効率が圧倒的に高い。
    - **標準化された統制**: 組織全体のセキュリティポリシーを一元管理し、新アカウント作成時も自動的に制限が適用される環境を実現する。
- **副業での需要**: ★★★★ (セキュリティ・コスト最適化)。「開発者の誤操作による海外リージョンでのリソース乱立」を防ぐ設定として、コスト意識の高い企業（月額30〜60万円〜）で非常に高く評価されるスキル。

## AWS Systems Manager Session Manager
- **概要**: 鍵管理やインバウンド通信の開放なしに、EC2等へブラウザやCLIから安全にアクセスできるマネージドサービス。
- **試験ポイント**:
    - **要件**: SSM Agentの稼働、および `AmazonSSMManagedInstanceCore` 権限を持つIAMロール（インスタンスプロファイル）の付与が必須。
    - **ネットワーク**: インバウンドポート（22や3389）の開放は不要。SSMエンドポイントへのアウトバウンド（HTTPS/443）接続のみ必要。
    - **監査**: セッション内容をS3やCloudWatch Logsへ記録可能。操作履歴をCloudTrailで管理できる。
- **実務での使い方**: 
    - **踏み台サーバー（Bastion）の廃止**: 踏み台管理の手間とセキュリティリスクを削減。プライベートサブネットのインスタンスにも、VPCエンドポイント経由等でセキュアに直接アクセス可能。
    - **トラブルシューティング**: 接続できない場合は「IAMロールの有無」「SSM Agentの稼働状態」「SSMサービスへのネットワーク疎通」をまず確認するのが定石。
- **副業での需要**: ★★★★★ (運用改善・セキュリティ向上)。「古い踏み台サーバーの刷新」や「リモートワーク適応のための安全なアクセス環境構築」として、5〜20万円程度のスポット案件から、中長期の保守案件（月30万円〜）まで幅広く需要がある。

## NAT ゲートウェイ
- **概要**: プライベートサブネット内のリソースが、インターネットへアウトバウンド通信を行うためのマネージドサービス。
- **試験ポイント**:
    - **送信元IPの統一**: 固定のElastic IPを持つため、外部からは全ての通信がNAT GWのIPに見える（宛先制限のある外部API連携等に必須）。
    - **構成**: パブリックサブネットに配置し、プライベート側のルートテーブルで `0.0.0.0/0` のターゲットにNAT GWを指定する。
    - **比較**: インターネットゲートウェイ（IGW）は各インスタンスにパブリックIPが必要なため、IP統一には不向き。
- **実務での使い方**: 
    - **可用性設計**: 1つのNAT GWは1つのAZに紐づくため、マルチAZ構成では各AZにNAT GWを配置するのがベストプラクティス（AZ間通信コストの削減にも繋がる）。
    - **コスト管理**: 時間課金に加え、データ処理量に応じた従量課金が発生するため、トラフィック量が多い場合はVPCエンドポイントの利用によるオフロードを検討する。
- **副業での需要**: ★★★★ (ネットワーク・コスト最適化)。「セキュリティ要件でプライベート配置にするが、外部連携が必要」という構成の相談は非常に多い。可用性とコストのトレードオフを説明できると信頼される。

## AWS Organizations と S3 バケットポリシーの連携
- **概要**: `aws:PrincipalOrgId` 条件キーを使用することで、S3バケットへのアクセスを特定のAWS組織（Organizations）に属する全アカウントのプリンシパルに限定する仕組み。
- **試験ポイント**:
    - **aws:PrincipalOrgId**: 個別のアカウントIDを羅列する手間を省き、組織ID（o-xxxxxxxxxx）一つで組織内からのアクセスを一括制御できる。
    - **ワイルドカードの活用**: `Principal: "*"` と条件（Condition）を組み合わせることで、「組織内の誰でも」という柔軟かつ安全なアクセス制御が可能。
    - **継承と自動適用**: 組織に新しいアカウントが追加された際、バケットポリシーを更新することなく自動的にアクセス権が適用される。
- **実務での使い方**: 
    - **中央ログ集約バケット**: 組織内の全アカウントからCloudTrailやVPC Flow Logsを1つのバケットに集約する際、管理工数を大幅に削減できる。
    - **データ流出防止**: 組織外のアカウントからのアクセスをAPIレベルで拒否するため、意図しない外部共有を防ぐ強力なセキュリティガードレールとして機能する。
- **副業での需要**: ★★★★ (セキュリティ・運用効率化)。「アカウントが増えてポリシー管理が限界」という企業に対し、管理コストを下げつつセキュリティを強化する「プロフェッショナルの仕事」として非常に高く評価される。

## Amazon CloudWatch Logs Insights
- **概要**: CloudWatch Logsのデータを専用のクエリ言語で高速に検索・分析できるマネージド型ログ分析サービス。
- **試験ポイント**:
    - **クエリ機能**: 最大20のロググループを横断的に検索可能。JSON形式のログは自動的にパースされ、フィールドとして抽出できる。
    - **比較対象**: リアルタイム異常検知なら `Contributor Insights`、大規模分析（データレイク）なら `Athena` だが、既存のLogsを即座にアドホック分析するなら `Logs Insights` が最適解。
    - **コマンド**: `filter`, `stats`, `sort`, `limit` などの主要コマンドの使い方（例：`stats count() by bin(5m)` で時間枠集計）。
- **実務での使い方**: 
    - **障害調査**: LambdaやECSのログから「特定のエラーが発生した回数を5分刻みで集計」や「特定のリクエストIDに紐づく一連のログを抽出」する際に非常に強力。
    - **運用効率**: インフラのセットアップが不要で、既存のログに対して即座にクエリを実行できるため、トラブル発生時のMTTR（平均復旧時間）短縮に直結する。
- **副業での需要**: ★★★★★ (監視・トラブルシューティング)。「原因不明のバグ調査」や「ログ分析基盤の構築」案件において、既存の構成を変更せずに即導入・成果を出せるため、信頼獲得に非常に有効な武器になる。

## CIS AWS Foundations Benchmark と AWS Security Hub
- **概要**: AWS環境のセキュリティ設定がベストプラクティス（CIS基準など）に準拠しているかを自動評価し、複数アカウントの結果を一元管理するサービス。
- **試験ポイント**:
    - **Security Hubの役割**: GuardDuty（脅威検知）、Inspector（脆弱性）などの検出結果を統合し、コンプライアンス（CIS、PCI DSS等）への準拠スコアを可視化する。
    - **Organizations連携**: 管理者アカウントから組織全体のアカウントを「自動登録」でき、新規アカウント作成時も即座に監視対象に含められる。
    - **CISベンチマーク**: IAM、ログ、ネットワークなどの具体的なチェック基準（ルートMFA、不要なキー削除等）が標準提供されている。
- **実務での使い方**: 
    - **セキュリティ専用アカウント**: Organizationsの「セキュリティ管理者」として特定のアカウントを指定し、全メンバーアカウントの準拠状況を中央集約して監視する。
    - **自動修復（Remediation）**: 非準拠を検知した際、EventBridge経由でLambdaを起動し、自動的に設定を修正（例：不要に開放されたSGを閉じる）するワークフローを構築する。
- **副業での需要**: ★★★★ (セキュリティ・ガバナンス)。「複数のAWSアカウントがあり、どこにセキュリティの隙があるか分からない」という企業に対し、短期間でセキュリティ診断と改善ロードマップを提示できる高単価なコンサル案件（月額40〜80万円〜）に繋がる。

## Route 53 ルーティングポリシーと段階的デプロイメント
- **概要**: Route 53は、トラフィックの配布方法を定義する多彩なルーティングポリシー（加重、レイテンシー、位置情報、フェイルオーバー等）を提供し、高可用性と戦略的なデプロイを可能にする。
- **試験ポイント**:
    - **加重（Weighted）**: 0-255の重みで比率配分。**カナリアリリース**や**A/Bテスト**のキーワードが出たらこれ。
    - **レイテンシー**: ユーザーから見て最も応答が速いリージョンへ誘導。グローバルパフォーマンス最適化に。
    - **位置情報（Geolocation）**: ユーザーの地理的場所（国・州）に基づく。コンテンツのローカライズや規制遵守（特定国のみ提供等）に。
    - **フェイルオーバー**: ヘルスチェックに基づき「プライマリ（正常時）→ セカンダリ（異常時）」へ自動切替。DR（災害復旧）対策の鉄板。
    - **複数値回答**: 最大8つの正常なレコードをランダムに返答。簡易的な負荷分散として機能。
- **実務での使い方**: 
    - **モダン・デプロイメントの心臓部**: CI/CDパイプライン（CodeDeployなど）と連携し、新バージョンの重みを「1% → 10% → 100%」と段階的に増やす**カナリアリリース**を自動化する。
    - **ブルーグリーンデプロイメント**: 加重ルーティングで新旧環境を「0:100」から「100:0」へ一気に、または徐々に切り替えることで、ダウンタイムをゼロにし、問題発生時の即時ロールバックを実現する。
    - **ヘルスチェックの徹底**: いずれのポリシーでも「正常なターゲットのみに飛ばす」ために、ELBやS3エンドポイント、外部URLのヘルスチェック設定を忘れずに行うのがプロの鉄則。
- **副業での需要**: ★★★★★ (CI/CD・DR設計)。「リリース時の不具合が怖くて手動作業に頼っている」現場に対し、Route 53を活用した自動化・安全なデプロイ戦略を提案・構築できるスキルは、開発プロセスの改善コンサル（月額50〜100万円〜）において極めて価値が高い。

## Amazon RDS バックアップとリードレプリカの活用
- **概要**: RDSのデータ保護（自動バックアップ）とスケーラビリティ・可用性向上（リードレプリカ）を組み合わせた、データベースの運用管理・復旧戦略。
- **試験ポイント**:
    - **自動バックアップ**: 日次の「自動スナップショット」と継続的な「トランザクションログ」の組み合わせで、**ポイントインタイム復元 (PITR)**（秒単位の復元）を可能にする。
    - **リードレプリカの昇格**: 読み取り専用のレプリカを独立したDBインスタンスへ「昇格」させて、書き込み可能なマスターとして利用できる。
    - **手法の比較**: 
        - **スナップショット復元**: 過去の「特定の時点」に戻したい時に使用。復元には新しいDBの起動時間がかかる。
        - **リードレプリカ昇格**: 「最新の状態」を維持したまま、障害時に即座に書き込み権限をバイパスしたり、テスト環境を切り出すのに有効。
        - **手動(pg_dump等)**: 運用負荷と時間がかかるため、マネージドな自動バックアップが第一選択肢。
- **実務での使い方**: 
    - **24時間以内のデータ復旧要件**: 「自動バックアップ（保持期間：1日以上）」を設定しておけば、最小限の運用負荷でPITRにより確実に要件を満たせる。
    - **昇格の活用シナリオ**: 読み取り不可になったマスターの代わりにレプリカを昇格させたり、本番データのコピーを即座に作って分析・開発環境として活用する。
    - **レプリカラグの監視**: 昇格時のデータ不整合を防ぐため、CloudWatchメトリクスでレプリケーション遅延を常に監視するのが鍵。
- **副業での需要**: ★★★★★ (DB運用・DR設計)。「DBのバックアップが正しく取られているか不安」「障害時に何分で復旧できるか定義したい」という企業の相談に対し、マネージドな機能を駆使して「最小の工数で最大の安心」を構築・マニュアル化できるスキルは、保守・運用案件（月30〜60万円〜）で非常に重宝される。

## Application Load Balancer (ALB) のメトリクス監視
- **概要**: ALBが自動的に収集・提供する標準メトリクス（RequestCount, TargetResponseTime等）を利用し、アプリケーションの健全性とトラフィック傾向を可視化する手法。
- **試験ポイント**:
    - **RequestCount**: ALBを通過した合計リクエスト数。追加設定なしで1分間隔のリアルタイム監視が可能。**Auto Scalingでインスタンスが増減しても、ALBレベルで一元的に集計**される。
    - **HTTPCode_Target_5XX_Count**: ターゲット（EC2等）側でのエラー数。4XX（クライアント側エラー）との切り分けが重要。
    - **TargetResponseTime**: ターゲットの応答速度。パフォーマンス劣化の早期発見に不可欠。
    - **比較**: 個別のEC2メトリクスでは「アプリ層の状況」が見えず、カスタムメトリクスは作成・保守コストがかかるため、まずはALBの標準メトリクスを活用するのが正解。
- **実務での使い方**: 
    - **ピーク分析とキャパシティプランニング**: 少なくとも2週間程度のトレンドを確認し、業務時間やスパイクの傾向を把握。これを基にAuto Scalingの閾値を微調整する。
    - **アラームの二段構え**: 5XXエラーの急増には即通知、RequestCountの異常な低迷には「サービス全体の死活」として通知するなどの多角的な監視をダッシュボード化する。

## Amazon Data Lifecycle Manager (DLM) とバックアップ自動化
- **概要**: EBSスナップショットやAMIの作成・保持・削除をスケジュールに基づいて自動化し、データの保護とコスト最適化を両立するマネージドサービス。
- **試験ポイント**:
    - **タグベースの管理**: リソースタグ（例：`Backup: Daily`）をキーに、ポリシーを一括適用可能。新規リソース追加時もタグ付与だけで自動的にサイクルに組み込まれる。
    - **クロスリージョンコピー**: DR（災害復旧）対策として、スナップショットを別リージョンへ自動コピーし、コピー先でKMSキーを用いた再暗号化も指定できる。
    - **暗号化の継承**: 元のボリュームが暗号化されていればスナップショットも自動暗号化される。コピー時にコピー先リージョンのキー（KMS）で再暗号化する設定が重要。
- **実務での使い方**: 
    - **運用管理の最小化**: カスタムスクリプトを一切使わず、DLMポリシーだけで「7日分保持」「月次は1年保持」といった複雑な要件を標準化し、運用エラー（削除漏れ等）を排除する。
    - **DR/コンプライアンス要件への対応**: 自然災害に備えた「地理的冗長化」を、追加のコード開発なしで実現。RTO/RPOの目標値に合わせたコピー頻度を設定する。
- **副業での需要**: ★★★★ (コスト最適化・ガバナンス強化)。「スナップショットの管理が放置され、不要なコストがかさんでいる」企業に対し、DLM導入によるコスト削減と安全なバックアップ体制の構築をセットで提案することで、即座に評価・リピートに繋がる（初期5〜20万円〜）。
