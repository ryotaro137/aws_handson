## VPCフローログとオンプレミス接続（VPN / Direct Connect）の考慮事項
- **概要**: AWSのVPCフローログにおいて、ハイブリッド環境（VPNやDirect Connect経由）のトラフィックを記録・分析する際の仕様と、オンプレミス環境のネットワーク機器ログとの特性・違いについての考慮事項。
- **試験ポイント**:
    - **記録範囲の限界**: フローログはIPパケット（L3/L4）のメタデータを記録するものであり、**VPN（IPSec）カプセル化のオーバーヘッドや設定・ステータス情報は記録されない**。同様に、DXの物理接続レベルの詳細やBGPルーティング情報、メトリクスも含まれない。
    - **カスタムフォーマットとサンプリング**: デフォルト項目（IP、ポート、バイト数など）に加え、カスタムフォーマットで追加フィールドを記録可能。また、すべてのパケットが記録されるわけではなくサンプリングが行われる点に注意。
    - **トラブルシューティング手順**: ログが出力されない場合、「ターゲットとフィルタ設定」「CloudWatch Logsへの配信設定」「配信を実行するIAMロールの権限」「サンプリング間隔・集約設定」を順に確認する。
- **実務での使い方**: 
    - **オンプレ機器とのログ突き合わせ**: オンプレミスのFWやスイッチ（物理層〜L7までキャプチャ可能）とVPCフローログ（L3/L4のフロー集約）を比較する際は、「バイト数の差異（カプセル化の影響）」「タイムスタンプのズレ」「集約レベルの違い」を前提に分析を行う必要がある。
    - **ハイブリッド環境の監視**: フローログだけではVPNやDXの「状態」は分からないため、AWSのネットワーク構成変更時はCloudWatchの VPN TunnelState や Direct Connect の ConnectionState メトリクスと組み合わせて監視する。
- **副業での需要**: ★★★★★ (ネットワーク設計・トラブル対応)。「オンプレからAWSへの通信が不安定」「通信量と課金の内訳を調査したい」といったハイブリッドネットワークのトラブルシューティングは難易度が高く、スポットでも高単価（10〜30万円〜）になりやすい重要なスキル。 

## Amazon S3 クロスアカウントレプリケーションと所有権の管理
- **概要**: 異なるAWSアカウント間でS3オブジェクトを自動複製する機能。レプリケーション先でのアクセス権限の取り扱い（オブジェクト所有権問題）と解決策が鍵となる。
- **試験ポイント**:
    - **所有権のデフォルト挙動**: デフォルト設定では、レプリケートされたオブジェクトの所有権は「ソース（送信元）アカウント」のままとなる。この状態では、宛先アカウントが自身のバケット内のオブジェクトにアクセスできない問題が発生する。
    - **Replica Owner Override (所有権のオーバーライド)**: レプリケーションルールの設定で `Change object ownership to destination bucket owner` を有効にすることで、宛先アカウントへ所有権を移譲できる。試験では「宛先でオブジェクトが読めない。どう設定すべきか？」というシナリオで頻出。
    - **必要な権限設定**: レプリケーションには双方向の許可が必要。ソース側にはレプリケート権限（`s3:ReplicateObject`等）を持ったIAMロール、宛先側にはバケットポリシーでソースアカウントからの受信を許可する設定が必須。また、双方のバケットで「バージョニング機能」を有効化しておく必要がある。
- **実務での使い方**: 
    - **バックアップとDR対策**: 運用アカウントから別組織（監査用、ログ保管用、バックアップ専用アカウントなど）へデータ保全を行う際に設定する。万が一送信元が不正アクセスを受けても、宛先アカウントが完全に分離されていればデータの損失を防げる。
    - **トラブルシューティング**: 「レプリケーションが動かない」という相談を受けた際は、1. バージョニングの有無、2. IAMロールとバケットポリシー（送受信の権限）、3. CloudTrailのエラーログ（アクセス拒否）、4. KMSの暗号化権限（暗号化されたオブジェクトの場合）の順に切り分ける。
- **副業での需要**: ★★★★ (コンプライアンス・バックアップ構築)。「別アカウントに監査用のログデータを安全に集約したい」「ランサムウェア対策として完全分離されたアカウントにデータを複製したい」といったセキュリティ・運用要件は大規模クライアントから非常に多く、IAM権限周りの複雑な設定をスムーズにこなせると評価が高い。

## AWS IAM Identity Center と Active Directory の統合
- **概要**: 複数のAWSアカウントやアプリケーションへのシングルサインオン（SSO）を提供するIAM Identity Center（旧AWS SSO）と、既存のMicrosoft ADを統合し、認証認可を集中管理する仕組み。
- **試験ポイント**:
    - **アイデンティティソース**: ユーザーとグループの情報源。既存のADをソースに指定することで、二重管理を防ぎ、パスワードポリシーやオンプレミスのID情報をそのままAWSでも利用可能になる。
    - **権限設定の構成要素**: 「権限セット（AWSリソースへのアクセス権限を定義したポリシー群）」と「アカウント割り当て（特定のAWSアカウントへのアクセス許可）」を組み合わせて管理する。
    - **グループベースのアクセス制御**: ADのグループ（例：`Domain Users`など）に対してIAM Identity Centerの権限セットを割り当てることで、ユーザー単位ではなくグループ単位で効率的にAWSのアクセス権を制御できる。
- **実務での使い方**: 
    - **ID管理の一元化**: 入社・退職時にAWS側のIAMユーザーを都度作成・削除する手間を省き、基盤となるADのユーザーを無効化するだけでAWSへのアクセスも即座に遮断できる。セキュリティやアカウント統制上非常に強力。
    - **組織の役割と権限セット**: 開発部にはフルアクセス権限、監査部には読み取り専用権限など、各組織の役割に応じた権限セットを用意し、ADグループにマッピングして運用するのがベストプラクティス。
- **副業での需要**: ★★★★★ (ID管理・セキュリティ統制)。「社員が増えてきて個別のIAMユーザー管理が限界」「オンプレのADと認証を統合したい」という情シス・コーポレートIT部門からの依頼が多い。AWS Organizationsと組み合わせたマルチアカウント基盤構築（30〜80万円〜）の要件として鉄板のスキル。

## Amazon EKS と MLを活用した高度な監視（CloudWatch / DevOps Guru）
- **概要**: コンテナ化されたEKS環境特有の動的なリソース変化に対応するための、コンテナ監視（Container Insights）と、機械学習（ML）を活用した異常検知の仕組み。
- **試験ポイント**:
    - **CloudWatch Container Insights**: EKSクラスター、ノード、ポッド、サービスレベルでのメトリクスとログを統合的に収集・可視化するコンテナ専用機能。従来のEC2レベルの監視ではポッドの挙動を追えないため必須となる。
    - **CloudWatch 異常検出 (Anomaly Detection)**: 静的なしきい値ではなく、MLによって過去のデータから正常なパターン（ベースライン）を学習し、動的なしきい値を作成する機能。季節性やトレンド（昼は高負荷、夜は低負荷など）を考慮できる。
    - **Amazon DevOps Guru**: 運用データをMLで分析し、異常の「検出」だけでなく「根本原因分析」や「具体的な修正アクションの推奨」まで自動で行うAI運用支援サービス。
    - **比較とユースケース**: デフォルトのCloudWatchアラーム（静的しきい値）は、常に使用量が変動するコンテナ環境では誤検知が多くなる。このような動的環境では「異常検出」や「DevOps Guru」を用いるのが正解ルート。
- **実務での使い方**: 
    - **アラート疲れの防止**: しきい値のチューニングを手動でやり続けるのは限界があるため、CPU使用率のアラームにCloudWatch異常検出の「ベースラインから○帯域外れた場合」という条件を設定することで、運用者の負担（深夜の無駄なアラート対応など）を劇的に減らす。
    - **トラブルシューティング**: 障害発生時、DevOps Guruのダッシュボードを見ることで「あのデプロイからDB接続エラーが増えている」といった関連イベントと根本原因のヒントを即座に得られる。
- **副業での需要**: ★★★★ (SRE・運用高度化)。「EKSを導入したものの運用が回っていない」「アラートばかりで狼少年状態になっている」という現場に対し、Container InsightsとML系監視を組み合わせた「スマートな監視基盤への刷新」を提案し、SREとしての価値を高めることができる。

## ロードバランサーと静的IP化（ALB / NLB / Global Accelerator）
- **概要**: AWSにおけるトラフィック分散と、各レイヤーでの「固定IPアドレス（静的IP）」の取り扱いや、アクセス最適化手法についてのアーキテクチャ設計。
- **試験ポイント**:
    - **ALB (Application Load Balancer / L7)**: HTTP/HTTPSの高度なルーティング（パスベースやホストベース）が可能。しかし、**IPアドレスは動的に変化するため直接Elastic IPを付与できない**（DNS名でアクセスするのが前提）。
    - **NLB (Network Load Balancer / L4)**: TCP/UDPに最適化され、超低レイテンシ・高スループットを誇る。最大の特徴として**Elastic IP（静的IP）を直接付与できる**。
    - **AWS Global Accelerator**: エッジロケーションを利用してグローバルな通信を高速化するサービス。**2つの固定されたエニーキャストIP（Anycast IP）**が提供される。これをALBの前段に配置することで、「ALBの高度なL7ルーティング機能」と「固定IPアドレス化」を両立できる。
- **実務での使い方**: 
    - **外部連携先への固定IP提示**: オンプレミス環境や外部の提携先企業から「FWの穴あけ（ホワイトリスト登録）のためにどうしても固定IPが必要だ」と言われた場合の定石。要件がL7（WAFやパスルーティング機能が欲しい）なら `Global Accelerator + ALB`、単純なL4のスループット重視なら `NLB` を選択する。
    - **DNSのキャッシュ問題の回避**: Route 53（DNS）を利用したルーティングだと、クライアント側（ブラウザやOS）のDNSキャッシュによってフェイルオーバーに時間がかかることがある。Global Acceleratorの固定IPを使えば、AWSネットワークの裏側でルーティングが切り替わるためダウンタイムを極小化できる。
- **副業での需要**: ★★★★ (ネットワーク設計・トラブル対応)。「クライアントからAPIの提供先として固定IPを要求されたが、ALBを使っているためどうすればいいか？」という相談は非常によくあるケース。即座にGlobal Acceleratorの提案ができると信頼感に繋がる（設計・構築：10〜20万円〜）。

## Amazon SNS による通知と CloudWatch / Webhook 連携
- **概要**: AWSのフルマネージドなプッシュ型通知サービスであるSNSを活用した、システム異常の迅速な検知と外部ツール（チケット管理システム等）への自動連携の仕組み。
- **試験ポイント**:
    - **CloudWatch アラームとの連携**: アラームの状態（OK, ALARM, INSUFFICIENT_DATA）が変化した際に、SNSトピックを直接ターゲットとして指定し通知を送ることができる。この時、アラームの詳細情報は自動的にJSON形式のメッセージとして配信される。
    - **ファンアウト（Fanout）アーキテクチャ**: 1つの出来事（メッセージ）を、複数の宛先（エンドポイント）へ同時にばらまくことができる機能。
    - **多様なプロトコルとWebhook連携**: メールやSMSだけでなく、HTTP/HTTPSプロトコルもサポートしている。チケット管理ツール（JiraやBacklogなど）のWebhook APIエンドポイントをSNSにサブスクライブ（登録）させることで、HTTP POSTリクエスト経由でアラーム情報を直接ツールへ送信可能。
- **実務での使い方**: 
    - **インシデント起票の自動化**: 深夜のエラーやシステム障害発生時、人間が手動でチケットを切るのではなく、`CloudWatch Alarm -> SNS (HTTPS) -> チケットツール` のように連携させ、アラームの内容（JSON）を元に自動で障害チケットを作成させるのが定石。
    - **Lambdaを挟んだデータ加工**: SNSからのJSONをチケットツールがそのまま解釈できない場合、間にLambda関数を挟んでペイロード（データの中身）をツールが読める形式に整形してからPOSTするアーキテクチャもよく使われる。
- **副業での需要**: ★★★★★ (運用自動化・SRE)。「エラー検知からチケット起票までを完全自動化したい」という要望は規模を問わず非常に多い。「運用者の負担を減らす自動化」は直接的な工数削減に繋がるため、単価交渉もしやすく継続案件（自動化パイプラインの拡充）に直結しやすい。

## Amazon CloudFront のキャッシュ制御と最適化
- **概要**: グローバルなエッジロケーションを利用してコンテンツ配信を高速化するCloudFrontにおいて、どのファイルをどのくらいキャッシュするか（TTL）を制御するメカニズムの理解。
- **試験ポイント**:
    - **TTL（Time To Live）の決定優先順位**: キャッシュ期間は、①「オリジンのHTTPヘッダー（Cache-Control等）」、②「CloudFrontディストリビューション設定（最小/最大TTL）」、③「デフォルトTTL」の順に評価される。
    - **Cache-Control ヘッダーの役割**: オリジンサーバが返す `Cache-Control` が最優先される。
        - `max-age`: 秒単位でのキャッシュ有効期間を指定。
        - `no-cache`: 「キャッシュしてもいいが、毎回使う前にオリジンへ新しいものがないか確認（検証）しろ」という指示。
        - `no-store`: 「絶対にキャッシュに保存するな」という厳格な禁止。
        - `must-revalidate`: キャッシュの期限が切れたら、必ずオリジンに検証に行くことを強制する。
- **実務での使い方**: 
    - **コンテンツに応じたキャッシュ戦略**: 
        - **静的コンテンツ**（画像、CSS、JSファイルなど）: 更新頻度が低いため、数ヶ月単位（例：`max-age=31536000`）の長期キャッシュを設定し、オリジンの負荷を極限まで下げる。
        - **動的コンテンツ**（APIレスポンス、ユーザー個別ページなど）: キャッシュしない（`no-store`）か、ごく短い期間（秒単位）だけキャッシュしてアクセス集中を凌ぐ。
        - **HTMLページ**: 頻繁に更新される可能性があるため、中期キャッシュや `no-cache` を使い、常に最新を返しつつ転送量を抑える。
    - **キャッシュ削除（Invalidation）の回避**: パスの末尾にバージョン番号を付与（`/v1/style.css`）することで、ファイルを更新した際にCloudFrontのキャッシュ削除（Invalidation: お金と時間がかかる）を待たずに即座に最新を配信するテクニックが定石。
- **副業での需要**: ★★★★ (パフォーマンス改善・コスト削減)。「Webサイトの表示が遅い」「EC2（オリジン）の負荷が高くて落ちる」「AWSの転送量費用が高い」といった悩みを抱えるクライアントに対し、CloudFrontの適切なキャッシュ設定（Cache-Controlチューニング）を行うことで、劇的な改善効果を提供できる。

## AWS Client VPN のトンネルモード構成
- **概要**: AWS環境やオンプレミスへ安全にリモートアクセスするためのマネージド型OpenVPNサービス。トラフィックの制御方法（トンネルモード）の設計が運用とコストに直結する。
- **試験ポイント**:
    - **フルトンネル（デフォルト）**: インターネット向けを含む**すべてのトラフィック**がVPNトンネルを経由してAWSへ送られる。一元監視が可能だが、不要なトラフィック（社員のYouTube視聴など）までAWSを経由するため、無駄な帯域幅の消費とコスト増加を招く。
    - **スプリットトンネル**: 特定の宛先（VPCのCIDR範囲など）への通信だけをVPNに流し、通常のインターネット通信は社員のローカル回線から直接アクセスさせる方式。帯域の節約とAWSのアウトバウンド通信コスト削減に非常に有効。
    - **変更方法**: Client VPNエンドポイントの「Modify Client VPN endpoint」から `Enable split-tunnel` を有効にし、VPNルートテーブルで対象となるVPC CIDRへのルートを追加することで機能する。
    - **証明書とDNS**: 相互認証にはクライアント・サーバー両方の証明書が必要だが、これはあくまで「認証」でありルーティングには影響しない。またDNSサーバの指定もVPC内の名前解決用であり、トラフィックをVPNへ流すかどうかの決定には関与しない。
- **実務での使い方**: 
    - **コストと遅延の削減**: 特別なセキュリティ要件（全通信をAWSのプロキシやFWで検査しなければならない等）がない限り、リモートワーク環境では**「スプリットトンネル」を有効化して構築するのが基本のベストプラクティス**。これによりVPNの通信遅延クレームとインフラコストの両方を抑えられる。
- **副業での需要**: ★★★ (リモートワーク環境構築)。「社員のテレワーク環境を安全に構築したい」「現在のVPNルーターが遅いのでAWSに移行したい」という要望に対するソリューション。スプリットトンネルの概念を理解して適切に設定することで、「VPNを入れたらWeb会議がカクカクになった」等のトラブルを未然に防げる。

## Amazon RDS のセキュリティとパラメーター管理
- **概要**: 高いセキュリティ要件（暗号化・アクセス制御）を満たすためのRDSの機能（SSL/TLS、KMS、セキュリティグループ）と、データベースエンジン設定（パラメーターグループ）の管理方法。
- **試験ポイント**:
    - **暗号化（転送時 vs 保存時）**: 
        - **転送時（SSL/TLS）**: クライアントとRDS間の通信を暗号化。パラメーターグループの `rds.force_ssl=1` を設定することでSSL接続を強制できる。
        - **保存時（KMS）**: ディスク上のデータをAES-256で暗号化。透過的に処理が行われ、CloudTrailで鍵の使用状況を監査できる。KMSは転送時の暗号化には関与しない。
    - **パラメーターグループ**: DBエンジンの設定群。`rds.force_ssl` のような「静的パラメーター」は変更後にDBインスタンスの再起動が必要。「動的パラメーター」は即座に反映される。デフォルトグループは変更不可のため、必ずカスタムパラメーターグループを作成する。
    - **アクセス制御（セキュリティグループ）**: IPアドレスや他のSGを指定してポートレベルのネットワークフィルタリングを行う。アプリケーションレベルの制御やデータの暗号化機能は持たない。
    - **拡張機能**: セキュリティや安定性を保つため、RDSでは事前承認された拡張機能のみが許可される。SSL/TLSは標準実装されているため、暗号化のために追加のカスタム拡張機能を入れる必要はない。
- **実務での使い方**: 
    - **コンプライアンス要件への対応**: 金融機関などでは「転送時と保存時の両方の暗号化」が必須となる。インフラ構築時にKMSの有効化と合わせて、カスタムパラメーターグループで `rds.force_ssl=1` を設定し、安全な通信を強制するのが定石。
    - **反映タイミング（再起動）の設計**: 静的パラメーターを変更する場合、いつ再起動してシステムを一時停止させるか（メンテナンスウィンドウ）の設計が必要になるため、運用への影響を必ず考慮する。
- **副業での需要**: ★★★★ (DBセキュリティ・監査対応)。「セキュリティ監査でRDSの通信が暗号化されていないと指摘された」という相談はよくある。カスタムパラメーターグループを用いたSSL強制化や、ダウンタイムを最小に抑える再起動計画の立案までリードできると非常に高く評価される。

## AWS における機密情報（シークレット）の管理戦略
- **概要**: データベースのパスワードや外部APIキーなどの機密情報を、ハードコード（ソースコードへの直書き）などの危険な手法から分離し、安全に保存・取得・運用するためのベストプラクティス。
- **試験ポイント**:
    - **AWS Secrets Manager**: 機密情報の管理に特化。最大の強みはRDSやAurora等のDBパスワードを**「自動ローテーション（定期更新）」**できる点。JSON形式での保存も可能。
    - **Systems Manager (SSM) Parameter Store (Secure String)**: パラメーター管理サービスだが、KMSと連携して暗号化保存（Secure String）が可能。階層構造（`/dev/db/password`など）で管理できる。自動ローテーション機能はないが、単純な構成用途ならSecrets Managerよりコストが安い。
    - **従来手法のリスク**: 
        - **ハードコーディング**: Github等での情報漏洩（最大のリスク）。
        - **ファイルベース/環境変数**: サーバー内での平文保存は、メモリダンプ等で漏洩する可能性があり実務・試験共にアンチパターンとされる。
- **実務での使い方**: 
    - **CI/CDパイプラインでの動的注入**: CodePipeline等でデプロイする際、ソースコード内には何も書かず、デプロイの実行時（またはコンテナの起動時）に一時的にIAMロールの権限を使ってSecrets Managerからパスワードを読み込み、環境変数としてアプリケーションに渡すのが現代のベストプラクティス。
    - **アクセス境界の多層防御**: 「SSMからパラメーターを読み取れるIAMポリシー（IDベースポリシー）」だけでなく、「VPCエンドポイント」を使ってインターネットを経由せずにAPIを叩く等、ネットワークレベルでも機密情報の流出経路を遮断する。
- **副業での需要**: ★★★★★ (DevSecOps・セキュリティ改善)。「GitHubにアクセスキーを上げてしまいAWSから警告が来た」といった緊急案件や、「ソースコードに直書きされている数百個のパスワードを安全な設計に直したい」というリファクタリング案件は単価も高く（50万円〜）、DevOpsエンジニアとして必須のスキルセット。

## CloudFront のヘッダー転送とデバイス最適化（User-Agent）
- **概要**: CloudFrontがオリジンへリクエストを転送する際、どのHTTPヘッダーを持たせて転送するか、またそのヘッダーを「キャッシュキー（キャッシュを分けるためのキー）」としてどう利用するかの設計。
- **試験ポイント**:
    - **デフォルトの挙動**: CloudFrontはパフォーマンス最適化のため、デフォルトでは `Host`、`Authorization`、`User-Agent` といった特定のヘッダーをオリジンに**転送しない**。
    - **ヘッダーベースのキャッシュ**: 転送するように設定したヘッダー（例：`User-Agent`）は、自動的にキャッシュキーの一部となる。つまり、「PCからのアクセス」と「スマホからのアクセス」で別々のキャッシュが生成されるようになる。
    - **キャッシュヒット率とのトレードオフ**: ヘッダーをたくさん転送（キャッシュキーに追加）すると、キャッシュのパターンが増えすぎて**キャッシュヒット率が低下する**。不要なヘッダーは転送しないのがベストプラクティス。
    - **オリジンカスタムヘッダー**: クライアントからのリクエストには含まれない独自の固定ヘッダー（パスワードやトークン等の静的な値）を、CloudFrontが付け足してオリジンへ送る機能。「この値を知っているCloudFrontからしかオリジン（ALB等）にアクセスさせない」というセキュリティ用途でよく使われる。
- **実務での使い方**: 
    - **レスポンシブWebデザインとデバイス検出**: `/page.html` という同一URLであっても、オリジン（Webサーバー）が `User-Agent` を判定してPC用とスマホ用で異なるHTML/画像サイズを返す場合、CloudFront側で `User-Agent` に基づくキャッシュ分割を設定しないと、スマホユーザーにPC用の大きな画像がキャッシュされて返ってしまう大事故が起きる。
    - **CloudFront-Is-*-Viewer の活用**: 単純に `User-Agent` をそのまま転送すると種類（iPhone, Android各種, Mac, Windows...）が多すぎてキャッシュ効率が落ちるため、CloudFrontが用意している `CloudFront-Is-Mobile-Viewer` や `CloudFront-Is-Desktop-Viewer` という「正規化されたフラグヘッダー」を転送してキャッシュ分割に使うのが定石。
- **副業での需要**: ★★★★ (パフォーマンス改善・トラブルシューティング)。「スマホで見ているのにPC版のサイトが表示される」「キャッシュが効いていない気がする」というトラブルシューティング案件でよく遭遇する問題。CloudFrontのBehavior設定とキャッシュキーの概念を深く理解していると即座に解決できる。
