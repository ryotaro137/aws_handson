## 学習内容

### CloudFormation テンプレートの構造とエラー検証
- **概要**: テンプレートは `Resources` セクション以外はすべてオプショナル。また、スタック作成前にサポート外のプロパティや無効値に対する構文検証が実施される。
- **試験ポイント**: `Parameters` や `Outputs` などのセクションは省略可能であることが問われる。また、検証エラーはリソース作成時の権限・不足エラーとは異なり「テンプレート定義自体の問題」として事前に発生する点が重要。
- **実務での使い方**: `Parameters` や `Outputs` は、環境（Dev/STG/Prd）ごとの再利用性や別スタックへの値引き渡しのために実務では積極的に活用する。事前に検証でエラーになる仕様を活かし、CI/CD上で `cfn-lint` 等を実行するシフトレフト手法が一般的。
- **副業での需要**: ★★★★★ (IaC / CloudFormation)。パラメータ化を適切に行うことで再利用性の高いテンプレートを設計できるスキルは、手動構築からのIaC移行案件等で極めて需要が高い。

### AWS::EC2::Instance のプロパティとVPC/サブネットの関係
- **概要**: EC2で自動生成される読み取り専用属性（PrivateDnsName等）は設定不可。また配置先の `SubnetId` を指定すると、所属するVPCも自動的に決定される。
- **試験ポイント**: 指定不可の読み取り専用属性（`InstanceId`, `PrivateIp`, `PrivateDnsName` 等）と設定可能なプロパティの違い。また `SubnetId` 指定時に矛盾するVPCを指定するとエラーになる引っ掛け問題に注意。NW設定は指定サブネットのVPC設定に従う。
- **実務での使い方**: リソース間の依存関係上「VPC等のNW基盤」と「EC2等のアプリケーション基盤」でスタック（テンプレート）を分離する設計がベストプラクティス。`PrivateDnsName` 生成はNW側の `enableDnsHostnames` / `enableDnsSupport` に依存するため基盤側の設定漏れに注意する。
- **副業での需要**: ★★★★★ (IaC / CloudFormation)。VPCやEC2など各種リソースの依存関係を正確に理解し、正しくエラーのないIaCコードを実装できる能力はすべてのAWS構築案件の基本であり必須要素となる。

### EFS のマウントターゲットとマルチAZ設計
- **概要**: EFSは複数AZにデータが自動複製・保護される。各AZのサブネット内にマウントターゲットを作成し、NFS（ポート2049）経由でアクセスすることでAZローカルの高速通信を実現する。
- **試験ポイント**: パフォーマンス最適化および通信コスト削減のため、EC2からは「同一AZ内のマウントターゲット」に接続する点が頻出。別AZは障害時のフォールバック先として機能し、パフォーマンスモード（General PurposeとMax I/O）の選択も問われやすい。
- **実務での使い方**: 複数AZでの可用性要件を満たしつつ、POSIX準拠の一貫性により複数インスタンスからの同時アクセス時の不具合を防ぐ設計とする。転送時暗号化（EFS Utils）や保存時暗号化の設定、NFSポート（2049）のSG許可など、セキュリティ面も含めたIaC構成が基本となる。
- **副業での需要**: ★★★★ (コスト最適化・アーキテクチャ設計)。複数サーバーでファイル共有するWebシステム案件においてEFSは定番。クロスAZ通信料金の発生やレイテンシー低下を避ける「同一AZマウントの原則」など、運用のボトルネックを考慮した構成を提案できると信頼感が高まる。

### CloudWatch Agent のカスタムディメンション機能
- **概要**: `amazon-cloudwatch-agent.json` の `append_dimensions` を用いて、収集するメトリクスに共通のディメンション（最大10個まで。静的文字列や `${aws:InstanceId}` などの動的変数）を付与できる。
- **試験ポイント**: カスタムディメンションによる環境・アプリ別の詳細なアラーム設定や分析が可能になる点が問われる。EventBridge（イベント処理）、CloudTrail（API監査）といった用途が異なるサービスとの引っ掛け問題に注意。エージェント設定変更後は再起動が必要となることも要確認。
- **実務での使い方**: 大規模環境の監視では、EC2 Systems Manager (SSM) の Run Command や State Manager を活用して、設定ファイルの一括適用とCloudWatch Agentの再起動を自動化するのが鉄則。適切なディメンション設計で、リソース増加時もダッシュボードやアラームをスケールしやすく構成する。
- **副業での需要**: ★★★★★ (監視・アラート設計)。OSレベルのメトリクス（メモリ・ディスク使用率など）やアプリログの収集はほぼ全案件で求められる。SSMと連携した効率的なデプロイ・運用手法が組めると、手運用を減らせるため現場で高く評価される。

### CloudFormation StackSets とグローバルリソースの展開
- **概要**: StackSetsで複数アカウント・リージョンに展開する際、更新が未適用のインスタンスは「OUTDATED」ステータスになる。S3などグローバルで一意な名前が必要なリソースは、全リージョンへの同時デプロイ時に名前重複で失敗する制約がある。
- **試験ポイント**: 「OUTDATED」ステータスになる条件や、S3バケット・IAM・証明書のDNS検証レコードなどのグローバルリソース制約に起因する非対称なエラーに関する問題が出やすい。また、エラー時の動作を制御する「失敗許容度（failure tolerance）」と「並列度（parallelism）」の活用も頻出。
- **実務での使い方**: テンプレート内にリソース名をハードコードせず、`${AWS::Region}` や `${AWS::AccountId}` などの動的変数をプレフィックスに付与して名前競合を回避する設計が必須（例: `mybucket-${AWS::Region}-${AWS::AccountId}`）。また、部分的な障害で全体運用が停止しないように失敗許容度を適切にチューニングする。
- **副業での需要**: ★★★★★ (IaC / CloudFormation)。マルチアカウント・マルチリージョン環境の統制（セキュリティ設定展開や標準化等）は需要が高単価になりやすい。グローバル制約の落とし穴を早期に予見し、スケーラブルで堅牢なテンプレートを設計・運用できれば一目置かれる。

### AMI IDのリージョン固有性と CloudFormation Mappings の活用
- **概要**: AMI IDは各リージョンで固有かつ異なり、他リージョンとは共有できない。テンプレートの `Mappings` セクションと `FindInMap` 組み込み関数を使用して実行時パラメータ（リージョンや環境）から動的に値を取得することで、単一テンプレートの可搬性を実現する。
- **試験ポイント**: AMIのクロスリージョンコピーを利用する際、コピー先では新しいAMI IDが発行される点や、複数リージョン環境でハードコーディングを避けるために `Mappings` と `FindInMap` を用いて動的に参照する可搬性設計のアプローチが問われやすい。
- **実務での使い方**: 環境（Dev/Stg/Prd）やリージョンに非依存なテンプレートにするため、設定値やAMI IDを `Mappings` セクションで一元管理（中央集権管理）する。変更の一貫性を保ち、手動介入なく同一テンプレートで自動化・複数展開できるようにするのがIaCのベストプラクティス。
- **副業での需要**: ★★★★★ (IaC / CloudFormation)。「環境ごとにテンプレートを分ける」という手動保守の問題を解決し、スケーラブルで保守性の高いクリーンなIaCコードを書くスキルは開発・移行案件で高く評価される。

### AWS Backup と関連ストレージ・データベースサービス
- **概要**: AWS Backupは、EBS、S3、RDS、DynamoDB、EFSなど複数AWSサービスのバックアップを一元管理・自動化するフルマネージドサービス。AWS Organizationsと連携し、マルチアカウントでのバックアップポリシーの統制も可能。
- **試験ポイント**: 各サービスごとに個別でスナップショットを管理するのではなく、「AWS Backupを用いてバックアッププラン（RPO/RTO）やボルトのアクセス制御を一元管理する」という統合管理のベストプラクティスが頻出する。クロスリージョンやクロスアカウントのバックアップコピーも重要。
- **実務での使い方**: 企業コンプライアンス要件を満たすため、AWS Organizationsと統合して全社のバックアップポリシー（タグベースでの自動適用など）を展開する。ランサムウェア対策として、バックアップを別アカウント・別リージョンのWORM（Write Once Read Many）対応ボルトに隔離する設計が鉄則。
- **副業での需要**: ★★★★ (セキュリティ・運用自動化)。手動や個別スクリプトのバックアップ運用を廃止し、AWS Backupへ一元化するリファクタリング案件は多い。データ飛びやランサムウェアに備えた「セキュアなバックアップアーキテクチャ」の提案は、顧客の安心感と直結するため重宝される。

### ステートフルアプリケーションとセッション維持（CloudFront・ALB）
- **概要**: ログイン状態などを持つステートフルなアプリでは、同一ユーザーの通信を同一インスタンスに結びつける必要がある。ALBの負荷分散アルゴリズム単体では実現できないため、Cookieを用いたセッション維持機能（スティッキーセッション）が必要となる。
- **試験ポイント**: CloudFrontのデフォルト挙動（Cookieを転送しない）とキャッシュヒット率低下のトレードオフ、およびALBのスティッキーセッション「Duration-based（ALB生成）」と「Application-based（アプリ生成）」の違いが頻出。ターゲット障害時には自動で別インスタンスへフェイルオーバーされる点も重要。
- **実務での使い方**: 本来はセッション情報をElastiCache(Redis等)へ外出しするステートレス化がAuto Scalingと相性が良いベストプラクティスだが、既存アプリの移行時にはALBのターゲットグループ設定でスティッキーセッションを有効化して対応することが多い。また、CloudFrontを通す場合は必要なカスタムCookie（JSESSIONID等）だけを転送・キャッシュキー評価するようにして、キャッシュ効率の低下を最小限に抑えつつセッションを維持する設計が求められる。
- **副業での需要**: ★★★★ (アーキテクチャ設計・移行)。オンプレミスにあった旧式（ステートフル）アプリケーションをリフト＆シフトでAWSへ移行する案件において、ALBやCloudFrontのCookie設定は超頻出の要件。ソースコード改修を避けつつスケーラビリティと可用性を維持するネットワーク設定力はインフラエンジニアの腕の見せ所となる。

### Amazon Aurora PostgreSQL のアーキテクチャとスケーリング
- **概要**: 1つのプライマリ(writer)と最大15個のAurora Replica(reader)で構成され、同一ストレージを共有し低遅延で同期される。用途に合わせたエンドポイント（クラスター / リーダー / カスタム）が提供される。
- **試験ポイント**: OLTP（更新系）とBI・分析（参照系）のワークロード分離が頻出。「クラスターエンドポイント」で書き込み、「リーダーエンドポイント」で読み取りを分散させるのが正解となる。読み取り負荷の増加に対しては、ダウンタイムを伴いコスト高な垂直スケーリング（スケールアップ）より、柔軟な水平スケーリング（reader追加＝スケールアウト）を優先する。
- **実務での使い方**: アプリやインフラフレームワーク側でDBの接続先を「更新用（writer）」と「参照用（reader）」に分割（CQRSパターン等の適用）する設計が基本。さらに特定業務（例: 重い分析バッチ）専用のreaderインスタンス群をカスタムエンドポイントとして分離することで、他業務への影響をなくす構成もよく使われる。
- **副業での需要**: ★★★★ (コスト最適化・アーキテクチャ設計)。RDBのパフォーマンス低下は典型的な課題であり、「リーダーエンドポイントを活用した読み取り負荷のオフロードと、それに伴う不要なwriterインスタンスのサイズダウングレード（コスト最適化）」は、費用対効果が高く提案しやすい。

### Enhanced Monitoring vs Performance Insights
- **概要**: Enhanced Monitoringは「OS層（CPU、メモリ等）」、Performance Insightsは「SQL層（DB負荷、スロークエリ）」の分析に特化したツール。監視のレイヤーが異なる。
- **試験ポイント**: 「OSプロセスの負荷特定」ならEnhanced Monitoring（CloudWatch Logsへ送信）、「待機イベントや重いSQLの特定」ならPerformance Insights（平均アクティブセッション: AASが指標）を選択する。両者の使い分けが極めて重要。
- **実務での使い方**: スロークエリが疑われる場合はPerformance Insightsでボトルネック（CPU待ち、ロック待ち等）を特定し、インフラ側のリソース不足（メモリ不足によるスワップ等）が疑われる場合はEnhanced Monitoringでプロセスごとのリソース消費を確認する。
- **副業での需要**: ★★★★★ (DBパフォーマンスチューニング)。単なる「サーバーが重い」という相談に対し、これらを駆使して「OS設定の問題か、アプリのクエリの問題か」を即座に特定できるスキルは非常に重宝される。

### ALB ヘルスチェックの失敗原因と対策
- **概要**: 最も多い原因は、ALBのターゲットグループで設定された「プロトコル・ポート」と、実際のアプリケーションの待受プロトコル・ポートの不一致。
- **試験ポイント**: ALBがHTTP(80)でヘルスチェックを試行しているのにアプリがHTTPS(443)のみで応答しているケースや、パス（例: `/`）に対して 200 OK 以外（404, 503等）を返している場合の切り分けが問われる。
- **実務での使い方**: ヘルスチェックパスは極力軽量なものを用意し、DB接続などが必要な重い処理は避けるのがベスト。また、セキュリティグループでALBからの通信が許可されているか、ターゲットのプロセスが稼働しているかを最初に確認する。
- **副業での需要**: ★★★★★ (監視・トラブルシューティング)。Webサイトの新規構築や移行で「ターゲットがHealthyにならない」というトラブルは非常に多く、この切り分けを迅速に行えるエンジニアは信頼される。

### S3 レプリケーションとクロスアカウントの所有権管理
- **概要**: バケット間でオブジェクトを自動・非同期にコピーする機能。クロスアカウント時、デフォルトでは送信元が所有権を保持するため、送信先で `AccessControlTranslation` 設定と `s3:ObjectOwnerOverrideToBucketOwner` 権限付与を行い所有権を譲渡するのが一般的。
- **試験ポイント**: クロスアカウントレプリケーションで「送信先からオブジェクトが見えない」トラブルに対し、`AccessControlTranslation` で所有権を上書き（Override）する設定が正解としてよく問われる。また、既存オブジェクトのコピーには「S3 Batch Replication」が必要な点も頻出。
- **実務での使い方**: ログ集約やバックアップ用アカウントへのデータ同期で使用。15分以内の着実な同期が必要な場合は「Replication Time Control (RTC)」を有効化する。既存の全データを同期したい場合はレプリケーション設定後にバッチオペレーションを実行する。
- **副業での需要**: ★★★★ (データ活用・バックアップ)。セキュリティ要件で商用環境とログ分析環境をアカウントごとに分けているケースが多く、権限トラブルの起きないクリーンな同期設定ができるスキルは重宝される。

### CloudFront ヘッダー転送と User-Agent によるキャッシュ分岐
- **概要**: CloudFrontはデフォルトで多くのヘッダーを削除するが、オリジンでのデバイス判定（モバイル/PC）等のために `User-Agent` などの転送設定が可能。転送されたヘッダーは自動的にキャッシュキーに含まれる。
- **試験ポイント**: ヘッダー転送（Forward Headers）を有効にすると、そのヘッダーの値ごとにキャッシュが生成される（Cache Variation）。特に `User-Agent` は種類が多いためキャッシュヒット率が劇的に低下するトレードオフの理解が問われる。
- **実務での使い方**: レガシーなサーバーサイド判定を行うアプリでは `Behavior` 設定で `User-Agent` を転送する。ただし性能面では、CloudFrontが自動付与する `CloudFront-Is-Mobile-Viewer` 等の専用ヘッダーをオリジン側で参照するのがベストプラクティス。また、全てのオリジンリクエストに認証情報を入れたい場合は「オリジンカスタムヘッダー」を使用する。
- **副業での需要**: ★★★★ (フロントエンド高速化・移行)。既存のサーバーサイドレンダリング(SSR)アプリをCDN化する際、「スマホ専用サイトが表示されない」といったトラブル対応や、適切なキャッシュ設計を行うために必須の知識となる。

### NAT ゲートウェイのサポートプロトコル制限
- **概要**: NAT ゲートウェイは TCP、UDP、ICMP のみをサポートする。マネージドで高可用だが、NAT インスタンスで対応可能だった特殊なプロトコル（IPSec、GRE、SCTP 等）は利用できない制約がある。
- **試験ポイント**: NAT インスタンスから NAT ゲートウェイへの移行時に「特定の通信（VPN用プロトコルやFTPアクティブモード等）が失敗する」シナリオでの原因特定として頻出。プロトコル制限は重要な切り分けポイント。
- **実務での使い方**: 一般的なWeb/DBサーバーのインターネットアクセスには NAT ゲートウェイを推奨。ただし、拠点間VPN（IPSec）をNAT越しに張る必要がある、または特殊な産業用プロトコルを使用する場合は、引き続き NAT インスタンス（または他アーキテクチャ）が必要になる可能性があるため、設計時のプロトコル確認を徹底する。
- **副業での需要**: ★★★ (インフラ移行・設計)。「最新のマネージドサービス（NAT GW）に移行したのに動かなくなった」というトラブルを、プロトコルレベルの知識で即座に解決できるとエンジニアとしての信頼が厚くなる。
