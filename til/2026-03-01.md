## 学習内容

## S3 ライフサイクルポリシー（不完全なマルチパートアップロードの破棄）
- **概要**: アップロード中に中断され、S3 上に残り続けている分割データを自動で削除する機能。
- **試験ポイント**: `AbortIncompleteMultipartUpload` アクションと、コスト最適化シナリオでの頻出度を把握する。
- **実務での使い方**: 全バケットに標準で「7日後に破棄」などのルールを適用し、意図しないストレージ課金を防ぐガードレールとして運用する。
- **副業での需要**: ★★★★☆。既存アカウントのコスト削減診断において、発見しやすく成果が出やすい（FinOps）ポイント。

## CloudFormation StackSets
- **概要**: 管理者が複数の AWS アカウントおよびリージョンにわたって CloudFormation スタックを一括展開・管理するサービス。
- **試験ポイント**: AWS Organizations との統合による「新規アカウントへの自動デプロイ」や、`FailureTolerance` などの並列・障害制御オプション。
- **実務での使い方**: 組織全体のセキュリティガードレール（共通 IAM ロール、Config 設定など）を 1 箇所から強制適用し、設定のばらつきを排除する。
- **副業での需要**: ★★★★☆。マルチアカウント環境の構築・運用案件で必須。各アカウントの手動設定をゼロにし、運用のオーバーヘッドを劇的に下げる提案が可能。

## VPC フローログの記録範囲と制限事項
- **概要**: VPC 内の ENI を通過する L3/L4 IP トラフィックメタデータを記録する機能だが、Amazon DNS や IMDS などの特定の内部トラフィックや VPN/DX のルーティング情報は記録されない。
- **試験ポイント**: **「記録されないトラフィック」**（Amazon DNS: 169.254.169.253, IMDS, パケットのペイロード）を把握しているかが問われる。「ペイロードの解析」が目的であれば、フローログではなく「VPC トラフィックミラーリング」を選択するのが正解となる。
- **実務での使い方**: オンプレミス側のリッチなパケットキャプチャ機器と比較して「L3/L4 のフロー情報（送信元・宛先・ポート・バイト数・ACCEPT/REJECT）」しか取れない点を理解し、ネットワークトラブル時に「SG や NACL で弾かれているか（REJECT）」の一次切り分けツールとして活用する。ログが配信されない場合は、対象リソースの IAM Role 権限不足を最初に疑う。
- **副業での需要**: ★★★☆☆。通信要件が厳しいエンタープライズ案件で必須。「通信できない！」と騒ぎになった際、「フローログにない＝そもそも AWS まで到達していない（オンプレや VPN の不具合の可能性）」などの切り分けや、AWS の仕様により**そもそも記録されない通信**であることを顧客に即座に説明できるとプロとして非常に頼りになる。

## AWS Client VPN のスプリットトンネル (Split Tunnel)
- **概要**: クライアント PC からの通信のうち、「AWS (VPC) 宛ての通信」だけを VPN 経由にし、「通常のインターネット向けの通信（Web ブラウジングなど）」は直接クライアントのローカル回線から逃がす設定。
- **試験ポイント**: **「データ転送コストの削減」**や**「VPN 帯域幅の最適化」**が正解のキーワード。スプリットトンネルが無効（フルトンネル）だと、ユーザーが YouTube などを見る通信もすべて AWS を経由してしまい、AWS からインターネットへ出ていく「アウトバウンドデータ転送料金」や「NAT ゲートウェイの処理費用」が跳ね上がる。
- **実務での使い方**: リモートワーク環境の構築において、デフォルトはフルトンネルになるため、特別な統制（すべての通信を社内プロキシや AWS 上のファイアウォールで検閲したいなど）がない限り、パフォーマンス向上とコスト削減のためにスプリットトンネルをオンにするのがベストプラクティス。
- **副業での需要**: ★★★★★。構築済みのリモートワーク用 VPN のコンサルティングにおいて、「AWS の通信費が高すぎる」という課題に対し、このチェックボックスを 1 つオンにするだけで劇的なコスト削減（FinOps）を達成できるマジックとなる。

## Elastic IP (EIP) と 0.0.0.0/0 (セキュリティグループ設定) の違い
- **概要**: EIP は自分のリソースの「固定住所（宛先）」、0.0.0.0/0 は「世界中の誰からでも（送信元範囲）」を指す概念。
- **試験ポイント**: セキュリティグループでの「最小権限の原則」を意識する。管理用ポート（22等）に 0.0.0.0/0 を使うのは NG。
- **実務での使い方**: HTTP(80) は 0.0.0.0/0 で公開するるが、外部サービスへの接続元制限や管理用アクセスには EIP を活用して固定化・限定化する。
- **副業での需要**: ★★★★★。セキュリティ不備（全開放）の指摘や、不要な EIP の削除によるコスト最適化は、案件の入り口として非常に有効。

## EC2 Image Builder
- **概要**: OS イメージ（AMI）の作成、管理、デプロイを自動化するマネージドサービス。最新のパッチ適用やセキュリティ設定済みの「ゴールデンイメージ」を継続的に作成できる。
- **試験ポイント**: **「AMI 作成の自動化」**や**「セキュリティパッチの最新化」**がキーワード。自作の Lambda やスクリプトで AMI を作成するよりも、Image Builder を使う方が管理負荷が低く、テストプロセスまで含めたパイプラインを構築できる点が正解の鍵。
- **実務での使い方**: 組織内で推奨される「標準 AMI」の作成パイプラインとして運用する。定期的なスケジュール実行や、ベース AMI の更新をトリガーに、必要なソフトのインストール、セキュリティスキャン、テストを経て、特定のアカウント群へ AMI を自動配布（共有）する。
- **副業での需要**: ★★★★☆。セキュリティコンプライアンスが厳しい案件で、「どうやって AMI を最新に保つか」という課題への回答として非常に強力。一度パイプラインを組めば運用が自動化されるため、保守性の高い提案として喜ばれる。
## EC2 インスタンスストア vs Amazon EBS
- **概要**: 物理直結で超高速だが一時的な「インスタンスストア」と、ネットワーク経由で永続的な「EBS」の使い分け。
- **試験ポイント**: **「データの永続性」**と**「コスト」**が最大の分岐点。インスタンス停止でデータが消えても良い「キャッシュ」や「一時ファイル」ならインスタンスストア、DB や OS 領域なら EBS を選ぶ。また、EBS の種類（gp3, io2, st1, sc1）ごとの性能特性（IOPS vs スループット）の使い分けが頻出。
- **実務での使い方**: 通常は gp3 を標準とし、極めて高い IOPS が必要な DB には io2 を検討する。ビッグデータ解析のワークロードでは、IOPS ではなくスループットが重要になるため st1 を選択するなど、ワークロードの特性に合わせてボリュームタイプを最適化（コスト削減）する。

### ストレージ比較表
| 特徴 | EC2 インスタンスストア | Amazon EBS (gp3/io2) | Amazon EBS (st1/sc1) |
| :--- | :--- | :--- | :--- |
| **接続方式** | 物理直結 (ホストサーバー) | ネットワーク経由 | ネットワーク経由 |
| **永続性** | **一時的** (停止/終了で消失) | **永続的** (独立して存在可能) | **永続的** |
| **主な性能指標** | 極低レイテンシー・超高 IOPS | 高 IOPS・低レイテンシー | 高スループット |
| **データタイプ** | SSD | SSD | HDD |
| **主な用途** | キャッシュ、一時ファイル | OS、DB、汎用ワークロード | ログ解析、ビッグデータ |
| **追加料金** | なし (インスタンス料込) | あり (容量・性能に依存) | あり (容量に依存) |

### ストレージパフォーマンスの要素
- **IOPS**: 1秒あたりの読み書き回数。トランザクションの多い DB で重要。
- **スループット**: 転送速度 (MB/s)。大容量ファイルの転送や動画配信で重要。
- **レイテンシー**: アクセス遅延。ユーザー体験に直結する。
- **ランダムアクセス**: 順不同なデータへのアクセス性能。キャッシュサーバーで特に重要。

- **副業での需要**: ★★★★★。既存環境の「とりあえず io2/io1 にして高くなっている」状況に対し、gp3 への移行や、一時データのインスタンスストア活用を提案することで、パフォーマンスを維持したまま劇的なコストカットを実現できる。

## AWS Global Accelerator (ALB 連携)
- **概要**: AWS のグローバルネットワークを使用して、ユーザーからアプリケーションへの経路を最適化し、低遅延と 2 つの固定 IP（Anycast IP）を提供するサービス。
- **試験ポイント**: **「Anycast IP（固定IP）」**と**「高速なフェイルオーバー」**がキーワード。Route 53 のレイテンシーベースルーティングとの違いとして、DNS キャッシュの TTL に依存せず、数秒以内に対象リージョンを切り替えられる点が重要。
- **実務での使いどころ**: 
  - **グローバル展開**: 世界中のユーザーに均一な低遅延を提供したい場合。
  - **IP 制限のあるクライアント**: クライアント側で接続先の IP をホワイトリスト登録する必要があり、ALB の IP 変化（スケーリングによる変化）を許容できない場合に AGA を前段に置く。

### Global Accelerator と ALB 連携のメリット
| カテゴリ | 特徴・メリット | 詳細説明 |
| :--- | :--- | :--- |
| **IP アドレス** | **固定静的 IP (Anycast)** | 2 つの不変な固定 IP を提供。FW 設定が容易。 |
| **ルーティング** | **最適な AWS リージョンへ** | AWS 専用網を通り、最も遅延の少ないリージョンへ誘導。 |
| **可用性** | **リージョン間フェイルオーバー** | リージョン障害時に数秒で他リージョンへ即時切り替え。 |
| **拡張機能** | **ALB の高度な制御** | AGA の固定 IP と、ALB のパス/ホストベースルーティングを両立。 |
| **パフォーマンス** | **TCP/UDP 高速化** | エッジロケーションでコネクションを終端し、AWS 網で高速転送。 |

- **副業での需要**: ★★★★☆。グローバル進出を狙うスタートアップや、古いオンプレミス思考で「固定 IP 接続」を絶対条件とするエンタープライズ案件において、モダンな AWS 構成とレガシーな要件を繋ぐ「神サービス」として提案価値が非常に高い。

## S3 クロスアカウントレプリケーションとオブジェクト所有権 (Replica Owner Override)
- **概要**: 異なる AWS アカウント間で S3 オブジェクトを自動複製する機能。複製先のオブジェクト所有権を「宛先アカウント」に自動変更（Override）することで、宛先側での完全な制御を可能にする。
- **試験ポイント**: **「所有者は誰か？」**が最大のひっかけ。デフォルトでは複製元（ソース）が所有者のままになり、宛先アカウントが自バケット内のデータを見れない事態が起きる。これを解決する **`Replica Owner Override`** オプションの選択が正解の鍵。また、ソース・宛先の**双方でバージョニング有効化**が必須である点も頻出。
- **実務での使い方**: セキュリティ監査やデータ集約のために、各部門アカウントのログを「ログ専用アカウント」の S3 に集める際に必須となる。IAM ロールと宛先バケットポリシーの両方で許可が必要。
  - **ソース側の権限**: `s3:GetObjectVersionForReplication`, `s3:ReplicateObject`
  - **宛先側の権限**: バケットポリシーでソースアカウントからの `s3:ReplicateObject` 受信を許可。

### レプリケーション設定の注意点
| 項目 | 必須要件・設定ポイント |
| :--- | :--- |
| **バージョニング** | ソース・宛先の**両方**で必ず有効化する。 |
| **所有権の転送** | `Change object ownership to destination bucket owner` を有効にする。 |
| **IAM ロール** | ソースバケットにアタッチし、宛先への複製権限を付与する。 |
| **既存オブジェクト** | デフォルトでは設定**以前**のオブジェクトは複製されない（バッチレプリケーションが必要）。 |

- **副業での需要**: ★★★★☆。マルチアカウント環境の設計（ランディングゾーン構築など）において、「各アカウントの CloudTrail ログを中央アカウントに安全にバックアップ・集約する」構成は定番。所有権の問題で「データはあるが中身が読めない」というトラブルを即座に解決できると非常に重宝される。

## Amazon Aurora Auto Scaling
- **概要**: 読み取り負荷（CPU 使用率や接続数）に応じて、Aurora レプリカ（リーダーインスタンス）の数を自動的に増減させる機能。
- **試験ポイント**: **「読み取り専用レプリカ（最大15台）のみ」**がターゲット。書き込みインスタンス（プライマリ）はスケーリング対象外。共有ストレージを使用するため、スケーリング時のデータ同期オーバーヘッドが極めて低い点が RDS 通常版との違い。
- **実務での使い方**: 読み取りエンドポイント（Reader Endpoint）を活用して、増減するレプリカに負荷を分散させる。スパイク的な負荷に備えて予備を置くのではなく、Auto Scaling でコストとパフォーマンスの両立を図る。また、AWS Auto Scaling（EC2, ECS, DynamoDB など）とは別個の機能である点に注意。

### Aurora DB クラスター構成と役割
| コンポーネント | 役割 | スケーリング |
| :--- | :--- | :--- |
| **プライマリインスタンス** | 書き込み (Writer) 担当。1台のみ。 | インスタンスタイプの変更 (垂直スケール) |
| **Aurora レプリカ** | 読み取り (Reader) 担当。最大15台。 | **Aurora Auto Scaling で自動増減。** |
| **共有ストレージ** | 全インスタンスでデータを共有。 | 自動拡張 (10 GB ごと、最大 128 TiB) |
| **読み取りエンドポイント** | リーダー群への負荷分散（DNS 解決）。 | スケーリングに合わせて自動管理。 |

- **副業での需要**: ★★★★☆。DB の高負荷による遅延改善や、常時稼働レプリカの削減によるコスト最適化案件で重宝される。

## RDS スナップショットのクロスアカウント共有と KMS 権限
- **概要**: RDS のスナップショットを別のアカウントに共有し、そこから復元（移行/複製）する手法。暗号化されている場合は、KMS キーポリシーの修正が必須となる。
- **試験ポイント**: **「暗号化されたスナップショットをどう共有するか？」**が超頻出。
  - **AWS 管理キー**で暗号化されたものは直接共有**不可**。
  - カスタマー管理キー (CMK) を使い、キーポリシーで移行先アカウントに `kms:Decrypt`, `kms:DescribeKey`, `kms:CreateGrant` を許可する必要がある。
  - 手順の順序（KMS 修正 → 共有 → 復元）も問われる。
- **実務での使い方**: アカウント統合や環境分割（Prod から Dev へのデータ同期）で利用。S3 エクスポートよりも高速で整合性が保ちやすい。

### クロスアカウント共有のプロセス
1. **スナップショット作成**: ソース側で取得（暗号化済み）。
2. **KMS キーポリシー修正**: 移行先アカウントの root ユーザーに対し、`kms:Decrypt` 等の権限を付与。
3. **共有設定**: RDS コンソールで移行先アカウント ID を指定。
4. **移行先で復元**: 共有されたスナップショットから新しい DB インスタンスを起動。

### なぜ他の選択肢（S3 / レプリカ）ではダメなのか？
| 比較項目 | RDS スナップショット共有 | S3 エクスポート/インポート | リードレプリカ (DR) |
| :--- | :--- | :--- | :--- |
| **手軽さ** | **高い**（マネージド機能のみ） | 低い（多くの手順が必要） | 中（クロスアカウント設定が複雑） |
| **コスト** | 低（保存料のみ） | 中（S3 料金 + 転送料） | 高（レプリカ稼働費） |
| **リードタイム** | 高速（ブロックレベル） | 低速（中間ファイル経由） | リアルタイムだが構築が重い |
| **主な欠点** | - | セキュリティリスク、手間 | 読み取り専用制限、ラグ |

- **副業での需要**: ★★★★☆。データの移行やバックアップ戦略の見直し案件で必須。「データが暗号化されていて共有・復元できない」という現場のトラブルを、KMS ポリシーの数行の修正で解決できるスキルは非常に重宝される。

## Amazon CloudFront のキャッシュ動作と TTL 制御
- **概要**: エッジロケーションでのキャッシュ保持期間（TTL）を、オリジンの HTTP ヘッダーやディストリビューション設定によって制御し、パフォーマンス向上とオリジン負荷軽減を図る仕組み。
- **試験ポイント**: **「キャッシュの優先順位」**と**「Cache-Control ヘッダーの挙動」**が最重要。
  - 基本は「オリジンからのヘッダー（Cache-Control 等）」が優先されるが、ディストリビューション側で設定した「最小 TTL / 最大 TTL」の範囲内に強制的に収められる点に注意。
  - `no-cache`（キャッシュするが使用前に検証必須）と `no-store`（一切キャッシュ不可）の違いなど、ヘッダーごとの意味が問われる。
- **実務での使い方**: 静的ファイル（画像・JS・CSS）は `max-age` を長く設定してキャッシュヒット率を高め、API レスポンスなどの動的コンテンツは TTL を 0 または極短時間に設定して整合性を保つ。更新頻度が高い HTML は「Invalidation（キャッシュクリア）」を CI/CD パイプラインに組み込む等の運用を行う。

### 主要な Cache-Control ヘッダー
| ヘッダー | 用途・挙動 |
| :--- | :--- |
| **max-age=[秒]** | キャッシュの有効期間を秒単位で指定。 |
| **no-cache** | キャッシュはするが、毎回オリジンに「更新がないか」の検証（再検証）を強制する。 |
| **no-store** | ブラウザや CDN に一切のキャッシュを保存させない。 |
| **must-revalidate** | 期限切れ（stale）になった後、オリジンでの検証なしに古いキャッシュを返させない。 |

### TTL 制御の優先順位と設定
- **優先順位**: オリジンヘッダー値が、CloudFront 側で定義した「最小 TTL 〜 最大 TTL」の範囲内であればヘッダーが採用される。ヘッダーがない、または範囲外の場合は「デフォルト TTL」や制限値が適用される。
- **ベストプラクティス**: 
  - 静的コンテンツ: 長期キャッシュ（数ヶ月〜 1 年）。
  - 動的・パーソナライズ: キャッシュ無効（TTL=0）。
  - HTML ページ: コンテンツ更新頻度に応じて中期キャッシュ（数分〜数時間）。

- **副業での需要**: ★★★★★。Web サイトの高速化（Core Web Vitals 改善）や、スパイクアクセスによるオリジンサーバー（EC2/RDS）のダウン防止策として、CloudFront のキャッシュ設計は極めて高いバリューを発揮する実績重視のスキル。

## AWS Config による設定監視と自動修復 (Remediation)
- **概要**: AWS リソースの設定履歴を記録し、定義したルール（コンプライアンス）に適合しているかを継続的に監視・評価するサービス。違反時に Systems Manager (SSM) Automation と連携して自動修復が可能。
- **試験ポイント**: **「継続的な設定監視」と「自動修復」**がキーワード。特に `s3-bucket-logging-enabled` ルール（S3 バケットのサーバーアクセスログ有効化）での違反検出と、SSM ランブックによる自動修正の組み合わせが頻出。
- **実務での使い方**: ガバナンスのガードレールとして、全バケットのログ有効化を強制。不適切な設定変更（ドリフト）を数分以内に自動で元に戻し、管理者に SNS 通知を飛ばす運用を構築する。

### AWS 周辺サービスとの比較
| サービス | 主な役割 | 強み | 弱点 |
| :--- | :--- | :--- | :--- |
| **AWS Config** | **設定**の履歴・関係性・適合性 | 状態の継続監視、**自動修復(SSM)** | 実行された「操作」の特定は不向き |
| **CloudTrail** | **API 呼び出し**の全記録 | 「誰がいつ何をしたか」の監査 | 現在の最新の「設定状態」把握は困難 |
| **CloudWatch** | **メトリクス・ログ**監視 | パフォーマンス異常検知、通知 | 設定内容が正しいかの評価はできない |
| **Trusted Advisor** | ベストプラクティス推奨 | コスト、セキュリティ等の改善提案 | 履歴の追跡や自動修復機能がない |

### S3 サーバーアクセスログの自動修復フロー
1. **検出**: AWS Config が S3 バケットの新規作成や設定変更を検知。
2. **評価**: `s3-bucket-logging-enabled` ルールで判定。ログ無効なら「非準拠」。
3. **トリガー**: 設定済みの SSM Automation (アクション: ログの有効化) が発火。
4. **修復**: 指定された S3 バケットにログ設定を強制適用し、準拠状態に戻す。

- **副業での需要**: ★★★★★。「人」のミスを「仕組み」でカバーするガバナンスの自動化は、特に高度な統制を求める企業のプロジェクト（月額 100 万円〜）やセキュリティコンサル案件で、専門家として非常に高く評価される。

## S3 Cross-Region Replication (CRR) と S3 Batch Replication
- **概要**: 異なる AWS リージョン間で S3 オブジェクトを自動・非同期で複製する機能。設定前に存在していた「既存オブジェクト」の複製には、オンデマンドの **S3 Batch Replication** が必須。
- **試験ポイント**: **「既存オブジェクトはデフォルトで複製されない」**点が最重要。設定前にあるデータ、Glacier 系のストレージクラス（※復元が必要）、所有権のないオブジェクトは自動複製の対象外。これらを複製するには `S3 Batch Replication` を使うのが正解となる。双方のバケットでの**バージョニング有効化**は前提条件。
- **実務での使い方**: ディザスタリカバリ (DR) やコンプライアンス要件（データの地理的冗長化）のために活用。過去分データの同期が必要な場合は、Batch Replication ジョブを作成し、進行状況を監視しながら実行する。

### CRR のレプリケート対象と制限
| 項目 | 挙動（デフォルト） | 注意点 |
| :--- | :--- | :--- |
| **新規/更新オブジェクト** | **複製される** | 設定後の作成・変更が対象。 |
| **既存オブジェクト** | 複製されない | **S3 Batch Replication** でジョブ実行が必要。 |
| **メタデータ / タグ** | 複製される | 属性情報も一貫して保持。 |
| **Glacier 系クラス** | 複製されない | 復元されている必要がある。 |
| **削除マーカー** | 複製されない | オプションで明示的な有効化が可能。 |

- **副業での需要**: ★★★★☆。BCP 対策（災害対策）の構築案件において、データの多重化は必須要件。既存の大量データを後から別のリージョンへ同期させる際の「Batch Replication」の活用方法と、それに伴うコスト（リクエスト料金等）を正しく管理できると非常に評価される。

## AWS Secrets Manager (パスワードの自動ローテーション)
- **概要**: データベースや API キーなどの機密情報を安全に管理するサービス。**「自動ローテーション」**機能により、Lambda と連携してパスワードを定期的に変更し、漏洩リスクを下げる。
- **試験ポイント**: **「パスワードの自動更新」**や**「Lambda との統合」**がキーワード。よくある比較対象の SSM パラメータストア（SecureString）との違いは「自動ローテーション機能の有無（Secrets Manager は標準であり、パラメータストアはなし）」が頻出のポイント。
- **実務での使い方**: RDS や Redshift、DocumentDB と直接連携し、メンテナンスウィンドウ中に自動で認証情報を更新する。アプリ側は最新のシークレットを API で取得する設計とし、パスワードの「ベタ書き（ハードコード）」を排除する。
- **副業での需要**: ★★★★★。高度なセキュリティ設計が求められる案件で必須。「単に隠す」だけでなく「定期的に変える」運用までセットで提案できると、信頼は絶大になる。
